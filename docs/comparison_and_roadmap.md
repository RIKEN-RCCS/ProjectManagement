# 一般的PMツールとの比較と今後の改善方針

作成日: 2026-03-25

---

## 本システムの有意な点（強み）

| 観点 | 内容 | 一般ツールとの差分 |
|---|---|---|
| **LLMによる自動抽出** | Slack スレッド・会議議事録（録音含む）から決定事項・アクションアイテムを自動抽出 | Jira/Asana/Monday は **全て手動入力**。本システムは入力コストゼロ |
| **決定事項の一元管理** | `decisions` テーブルで決定事項を独立管理。Slackパーマリンク・議事録パスで背景に遡れる | 一般ツールでは決定事項はコメントやノートに埋没し、検索・追跡が困難 |
| **2層構造（TD/BU統合）** | goals.yaml（人間が承認・Gitで変更履歴管理）↔ Slack/議事録からの自動抽出を統合 | 一般ツールはBU（タスク入力）のみ。「今プロジェクトがどこにいるか」を答えられない |
| **Slack Canvas Native統合** | 別ツールへの移行不要。Canvas上で直接編集→pm.dbに同期 | Slack連携はあるが、Canvasネイティブ双方向同期は存在しない |
| **変更履歴の完全追跡** | audit_log（Canvas同期・マイルストーン紐づけの前後）+ Git（goals.yaml変更） | Jiraはフィールド変更履歴あるが、外部ツール（Canvas）との整合は取れない |
| **会議録音→PM自動化** | mp4/m4a → Whisper文字起こし → pm.db まで全自動（Slurm） | 同等機能を持つPMツールは存在しない |
| **差分処理による効率化** | 変化のないSlackスレッドはAPI・LLM呼び出しゼロ | SaaS型ツールは常時フル再取得 |
| **セキュリティ** | SQLCipher暗号化・トークンファイル分離・平文ファイル削除（--meeting-name） | SaaSは全データを外部サーバに預けるリスク。本システムはオンプレミス完結 |

---

## 本システムが劣っている点（弱み）

| 観点 | 内容 | 一般ツールでは |
|---|---|---|
| **自動定期実行がない** | Slack取得・抽出・レポート投稿が全て手動コマンド | Jira/Asana/Monday は Automation ルールが標準搭載 |
| **非技術メンバーへの敷居** | CLI操作・CSV経由編集が必要。Canvas上での編集は可能だが機能が限定的 | Web UI で誰でも操作可能 |
| **goals.yaml が未作成** | トップダウン層（マイルストーン管理）の仕組みは実装済みだが、データが存在しない | 一般ツールではプロジェクト作成時にマイルストーン設定が必須フロー |
| **外部ステークホルダー追跡がない** | NVIDIA・富士通へのアクションアイテムが組織単位で追跡できない | Asana/Monday はゲストアクセス・外部連携が標準 |
| **ガントチャート・依存関係がない** | タスク間の依存関係定義・視覚的タイムライン表示なし | Jira/Asana/Monday は全て標準搭載 |
| **通知・リマインダーがない** | 期限切れアイテムは pm_report.py 実行時にのみ把握可能 | 一般ツールは期限前・期限超過の自動通知 |
| **マルチプロジェクト非対応** | pm.db は単一プロジェクト前提。goals.yaml も1ファイル | Jira は複数プロジェクト・ポートフォリオ管理が標準 |
| **LLM抽出品質の保証がない** | 誤抽出・漏れ抽出の検知・バリデーション機構がない | 手動入力前提のため別の問題 |

---

## 現状維持で許容できる差分

以下は一般ツールとの差分だが、本プロジェクトの文脈では **対応不要**:

- **ガントチャート**: 富岳NEXTはハードウェア制約主導のスケジュールで、タスク間依存よりもマイルストーン達成の追跡が本質。goals.yaml + pm_report.py で代替可能。
- **マルチプロジェクト対応**: 本システムはアプリケーション開発エリアに特化した設計。複数プロジェクトへの汎化はスコープ外。
- **リソース管理・予算追跡**: エリアマネージャの職掌ではなく部門長・マネジメント室の領域。PMBOKの「リソース管理」は本システムの対象外。

## 今後の改善方針（優先度順）

### P0（即時・1-2日）: goals.yaml の作成とマイルストーン稼働開始

**問題**: トップダウン層が未稼働。pm_report.py のマイルストーンセクションが空欄のまま。

**対応**:
1. `goals.yaml` を作成し、現在承認されているゴール・マイルストーンを記述
2. `pm_goals_import.py` で pm.db に同期
3. `pm_relink.py --export` で既存アクションアイテムを milestone_id に紐づけ（CSV編集）

**効果**: 本システムの設計思想（2層構造）が初めて稼働する。レポートに「プロジェクトの現在地」セクションが出力される。

---

### P1（短期・1週間）: 自動定期実行の設定

**問題**: 全スクリプトが手動実行のため、最新情報反映に運用者の積極的な操作が必要。

**対応**: Slurm `--cron` または OS `crontab` で以下を設定

```sh
# 平日朝8時: Slack取得・抽出
0 8 * * 1-5  bash ~/ProjectManagement/slack/scripts/pm_from_slack.sh -c C08SXA4M7JT

# 月曜朝9時: 週次レポート生成・Canvas投稿
0 9 * * 1    bash ~/ProjectManagement/slack/scripts/canvas_report.sh
```

**効果**: 運用負荷の大幅削減。毎朝最新状態が Canvas に反映される。

---

### P2（短期・1週間）: 外部ステークホルダー追跡の強化

**問題**: NVIDIA・富士通への依頼事項が個人担当者に埋もれ、組織単位の「外部待ち」が可視化されない。

**対応**:
1. `action_items.assignee` に組織名（`NVIDIA` / `富士通`）を入力する運用ルールを CLAUDE.md に追記
2. `pm_report.py` の「要注意事項」セクションに「外部待ちアイテム」自動抽出ロジックを追加（`assignee IN ('NVIDIA', '富士通')` かつ status='open'）
3. `pm_insight.py` の LLM プロンプトに「外部待ちアイテム」統計を追加

**効果**: 外部依存によるプロジェクト停滞を早期に可視化できる。

---

### P3（中期・2-4週間）: LLM抽出品質の向上

**問題**: 誤抽出・漏れ抽出を確認するには pm_relink.py でCSV全件確認が必要で手間がかかる。

**対応**:
1. `pm_extractor.py` / `pm_minutes_import.py` に「低信頼度フラグ」を実装（LLMに確信度を返させる）
2. `pm_report.py` の要注意事項に「未確認・低信頼度アイテム」セクションを追加
3. Canvas 上の行削除を検知して pm.db から soft delete する仕組みの検討

**効果**: LLM誤抽出の修正コストが下がり、データ品質が向上する。

---

### P4（中期・1ヶ月）: Slack App Socket Mode UIの実装

**問題**: 非技術メンバー（管理室・リーダー陣）が CLI/CSV 操作できない。

**対応**: `SLACK_APP_TOKEN` (xoxb-) を追加取得し、Socket Mode で Slack App を実装

優先実装コマンド:
- `/pm-close [ID]` — アクションアイテムをクローズ
- `/pm-assign [ID] [担当者]` — 担当者を変更
- `/pm-report` — 週次レポートを手動生成・Canvas投稿
- `/pm-sync` — Canvas → pm.db 同期実行

**効果**: テクニカルでないメンバーがSlack上でPMシステムを操作可能になる。

---

### P5（長期・要検討）: 達成証跡の登録（pm_document_import.py）

**問題**: success_criteria（「報告書の提出」等）に対して実際の資料を紐づける手段がない。マイルストーン「達成済」の自動判定ができない。

**対応**: `documents` テーブルと `pm_document_import.py` の実装

**優先度**: P4以降。goals.yaml + マイルストーン管理が安定稼働してから判断する。
