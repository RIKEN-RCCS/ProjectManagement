# KDDI NVL72 性能測定：go/no-go 意思決定メモ

- 決定日: 2026-07-01
- ステータス: 条件付き GO
- 関連: 富岳NEXT 将来計算機仕様検討

## Context（なぜこのメモか）

今秋に予定している KDDI 保有 NVL72 でのアプリ性能測定について、測定の「目的」が曖昧化し、そもそも実施の意義が見えなくなっているという相談から始まった。整理すると論点は以下：

- 手持ちに NVL72 を使いこなすほどチューニング済みのアプリがほぼ無く、FFT 基礎性能 / Gromacs 等 Proxy アプリを候補にしている。
- 候補メニュー：① NVL72 までの NVLink スケーラビリティ、② NVL4 相当を IB で束ねたスケールアウトのスケーラビリティ。だが積極的に測る理由が見つからない。
- 理研 Rikyu が既に「NVL4 を IB で複数繋いだ構成」を持つため、Rikyu で出来ることに高額の KDDI 利用料を払う価値がない。
- 本来は将来計算機（富岳NEXT）の仕様決定に資する測定であるべき。それが実現できるのか、意義があるのかを判断したい。

**確定した前提（本人確認済み）**：
- KDDI と Rikyu の差は **「NVLink ドメインサイズのみ」**（両者ほぼ同世代 GB200 相当）。**72-GPU 単一 NVLink ドメインだけが KDDI 固有**。
- 富岳NEXT の未決定の設計問題は **「密結合コヒーレントドメインのサイズ」** ＋ **「スケールアウト網の設計」** の 2 つ。

**意図する結果**：KDDI で「何を測る／測らない・なぜ」を根拠付きで確定し、無意味な高額測定を避けつつ、富岳NEXT 仕様決定に効く測定だけに絞り込む。

## 核心の問い

測定メニューを「能力の名前（スケーラビリティ）」で立てる限り、常に「Rikyu でもできる」に突き当たる。立てるべき問いは 1 つだけ：

> **KDDI でしか取れず、かつ富岳NEXT の未決定を動かすデータは何か。**

## 判断の骨格

### 差別化要素はただ 1 つ：72-GPU 単一 NVLink ドメイン

世代差がないため、KDDI（4→72 を NVLink 内）と Rikyu（4 は NVLink 内、それ以上は IB 跨ぎ）の差は **純粋に「NVLink ドメインをどこまで広げたか」だけに帰属できる**。交絡のないクリーンな制御実験になる。

逆に、Rikyu で同じ数字が出る測定（単純スケーラビリティ曲線、絶対性能ベンチ、IB スケールアウト単体）は **定義上 KDDI で測る価値ゼロ**。全てメニューから外す。

### 2 つの未決定は結合した 1 つの測定で答えられる

ドメインサイズと scale-out 網設計は独立でない。**大きな NVLink ドメインは、同じ総 GPU 数を達成するのに必要な IB 段数／二分割帯域を減らす**（通信の多くが NVLink 内に閉じるため）。よって「ドメインを 72 まで広げると scale-out 網への要求がどれだけ緩むか」を測れば、両方の問いに同時に効く。

### 測定は「絶対値」でなく「Rikyu との差分実験」として設計する

意義があるのは以下の差分だけ：
- 同一ワークロードを **KDDI で 4→8→16→36→72 GPU を NVLink ドメイン内でスイープ**
- **Rikyu で同数を、4 を超えたら IB 跨ぎでスイープ**
- 2 本のスケーリング曲線の **デルタ ＝ NVLink ドメインを 4 から 72 に広げた正味の価値**

## 測定メニュー（GO 部分の設計骨子）

原則：**「Rikyu で回したら結果が変わるはずのもの」だけ**を KDDI で回す。

| 測定 | 何を見るか | KDDI固有価値 | 優先度 |
|---|---|---|---|
| NCCL コレクティブ microbench（all-to-all / allreduce / allgather の帯域・レイテンシを 4..72） | 相互結合そのものの素の特性。差分実験の基礎 | ◎ ドメイン拡大の生の効き | 最優先・低コスト |
| 3D FFT 強スケーリング（cuFFTMp / heFFTe / P3DFFT を 4..72） | 全対全＝グローバル転置、bisection-bandwidth 律速。NVLink vs IB の差が最も鋭く出る | ◎ 主砲 | 最優先 |
| (条件付き) 大ドメイン×少数IB vs 小ドメイン×多数IB（同一総GPU数で比較、詳細は次項） | ドメインサイズが scale-out 網要件をどう緩めるか | ○ KDDI が 72 超構成を持つ場合のみ | 依存確認後 |
| Gromacs 等 Proxy | 近傍ハロ交換中心・比較的 compute bound。ドメインサイズ感度が低い | △ Rikyu でも近い数字 | 補助・任意（主目的でない） |

Gromacs は「主砲」ではない。ハロ交換はドメインサイズに鈍感で KDDI 固有価値が薄い。回すなら「compute-bound 側の対照群（ドメインを広げても差が出ないことの確認）」として位置づけ、汎用ベンチの素回しにはしない。

### 「大ドメイン×少数IB vs 小ドメイン×多数IB」で評価すべきポイント

この比較で決めたいのは「どちらが速いか」ではなく、**ドメインサイズへの投資と IB 網への投資がどれだけ代替可能か**という定量的な trade-off 曲線。前提として KDDI が NVL72 を複数台持つ場合のみ成立する（単一構成なら不成立、依存事項1を参照）。

1. **トラフィックの局所性**：同一通信パターン（特に FFT の全対全転置）で、総データ移動のうち何 % が NVLink 内に閉じ、何 % が IB を跨ぐか。ドメイン拡大の効果を最も直接示す一次指標。
   - 計測: `ibstat` / `perfquery`（IB バイトカウンタ）と `nvidia-smi dmon` / NVML（NVLink カウンタ）を collective 実行前後で差分取得。Nsight Systems の NCCL 計装タイムラインでも可視化可。
2. **Collective 性能の IB 帯域依存度**：NCCL は階層アルゴリズム（ノード内 → ノード間）を使うため、ドメインサイズが変わるとノード間ステップに乗るメッセージ数・サイズが変わる。ここが本丸。
   - 計測: `nccl-tests`（`alltoall_perf` / `allreduce_perf`）を同一総GPU数・異なるドメイン分割で回し、達成バス帯域（busbw）と時間を比較。IB 帯域は一定に保ち、ドメインサイズだけを変数にする。
3. **同一総GPU数でのアプリ強スケーリング**：固定 N に対し、大ドメイン少数IB vs 小ドメイン多数IB で FFT の time-to-solution / strong-scaling 効率を比較。ドメイン境界を跨ぐ全対全転置がどれだけボトルネック化するかを見る。
4. **代替曲線（本命の成果物）**：1〜3 の測定点から「ドメインサイズを D から D/2 に落としたとき、同じ collective 時間を維持するには IB 帯域を何倍にする必要があるか」という等性能等高線を導出する。「大ドメイン1単位の投資は IB帯域◯Gbpsの増設と等価」という形で言語化できれば、コスト比較（NVLink Switch 増設 vs IB 増強のどちらが単位性能あたり安いか）に直結する。
5. **コントロール**：同じ分割パターンで Gromacs を回し、曲線がフラットであることを確認する。差が出ないことが「効果は通信パターン（bisection-bound かどうか）依存である」という主張の裏付けになる。

## 成功基準 ＝ 富岳NEXT 判断への写像（測定前に人間が閾値を決める）

デルタが出たら何を言えるのかを **事前に** 定義しておかないと「測って終わり」になる。例：
- FFT 強スケーリング効率が **72 GPU で X% 維持**、一方 Rikyu 型（IB 跨ぎ）では **N GPU で Y% に頭打ち**、その差が閾値 Z 以上 → 富岳NEXT は大コヒーレントドメインへの投資が正当化される。
- 逆にデルタが小さい（IB スケールアウトでも十分追随） → 小ドメイン＋強い scale-out 網の方が費用対効果が高いという設計指針。
- scale-out 比較：大ドメイン構成で同一総 GPU 数を IB 二分割帯域あたり◯% 少ないコストで達成できる → 網設計の緩和余地を定量化。

X／Y／Z の具体値は、富岳NEXT で想定する total GPU 規模と対象ワークロードの compute/comm 比から逆算して事前設定する。

## 決定：条件付き GO

- **GO**：「NVLink ドメインサイズ感度の差分実験」（NCCL コレクティブ＋3D FFT を主砲、Rikyu を baseline）。KDDI でしか取れず、富岳NEXT の 2 つの未決定を直接動かす。
- **NO-GO（メニューから除外）**：Rikyu で同じ数字が出る全て——単純スケーラビリティ曲線、絶対性能ベンチ、IB スケールアウト単体、Proxy アプリの汎用素回し。
- **撤退線（測定自体をやめる条件）**：次のすべてが成立するなら KDDI に払う意義は消える：
  1. KDDI が単一 NVL72 しか無く scale-out 比較ができず、かつ
  2. pre-flight 見積（Rikyu 上の FFT/NCCL 通信プロファイルからの外挿）で 72 までのデルタが閾値 Z に届かないと予想され、かつ
  3. 富岳NEXT の対象ワークロードが compute-bound 寄りでドメインサイズに鈍感。

## 事前に確認すべき依存事項（測定計画確定の前提）

1. **KDDI の NVL72 は何台構成か** — scale-out 比較（2つ目の問い）が測れるかの前提。単一なら scale-out メニューは落ちる。
2. **同一 FFT 実装（cuFFTMp / heFFTe）が両機で動くか** — 秋までに両機でビルド・実行できる目処。
3. **Rikyu baseline を先に取得できるか** — KDDI は高額なので、Rikyu で曲線と pre-flight 見積を先に完成させ、KDDI は「72 固有の点を埋めるだけ」に絞ると費用対効果が最大化する。

## 補足：構造化台帳との対応

本判断は pm.db の意思決定台帳（`decisions` テーブル、`ledger_*`）にそのまま写像できる形にしてある。

- `trade_off`: 大 NVLink ドメインへの投資 vs scale-out 網強化への投資
- `reversal_condition`: 上記「撤退線」の 3 条件
- 台帳への投入は任意（`data/ledger_seed.json` に追記 → `scripts/ingest/ledger.py` で pm.db へ取り込み）。実施していない場合、このファイルが一次情報源。
